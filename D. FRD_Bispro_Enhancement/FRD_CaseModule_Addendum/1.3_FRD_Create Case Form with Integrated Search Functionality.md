# Functional Requirements Document (FRD)
## Create Case Form with Integrated Search Functionality

**Document Version:** 1.0  
**Last Updated:** June 2026  
**Status:** Draft

---

## 1. Overview

This document outlines the functional requirements for enhancing the **Create Case** form to include intelligent, interconnected search functionality for Account, Contact, and Asset fields with database integration.

### 1.1 Purpose
Enable users to efficiently create cases by:
- Searching and selecting Accounts from the database
- Dynamically filtering and selecting Contacts associated with the selected Account
- Searching and selecting related Assets from the database
- Implementing a command palette (Command Search) for quick access to these features

### 1.2 Scope
- Account search field with autocomplete
- Contact field with Account-dependent filtering
- Asset field with Account-aware search
- Command palette integration for quick searches
- Database query integration for all three entities

## 2. User Stories & Requirements

### 2.1 Account Search Field

**User Story:**
> As a case creator, I want to search for and select an Account so that I can associate the case with the correct organization/customer.

**Functional Requirements:**

| Requirement ID | Description | Priority |
|---|---|---|
| FR-101 | Account field shall display a searchable input with placeholder text "Search Accounts..." | HIGH |
| FR-102 | As user types, system shall query database and display matching Accounts in real-time (debounced) | HIGH |
| FR-103 | Search shall match Account names, ID numbers, and any other indexed fields | HIGH |
| FR-104 | Search results shall display Account name and relevant metadata (e.g., Account ID, type) | HIGH |
| FR-105 | User can select an Account from the dropdown results | HIGH |
| FR-106 | Selected Account shall be highlighted and locked in the field | HIGH |
| FR-107 | User can clear the selection via an "X" button or by deleting the value | MEDIUM |
| FR-108 | Account field is marked as required (*) on the form | HIGH |
| FR-109 | Search shall support pagination or "Load More" if results exceed 10 items | MEDIUM |
| FR-110 | Search shall be case-insensitive and handle special characters | MEDIUM |

**Design Details:**
- Input type: Autocomplete dropdown with search
- Minimum characters to trigger search: 1
- API debounce delay: 300ms
- Max results shown initially: 10
- Field state: Required (red asterisk)
- Visual feedback: Loading spinner during search

---

### 2.2 Contact Field (Dependent on Account Selection)

**User Story:**
> As a case creator, after selecting an Account, I want to search for and select a Contact associated with that Account so that the case is linked to the right person.

**Functional Requirements:**

| Requirement ID | Description | Priority |
|---|---|---|
| FR-201 | Contact field shall only be enabled after an Account is selected | HIGH |
| FR-202 | Contact field shall display placeholder "Search Contacts..." when enabled | HIGH |
| FR-203 | When enabled, Contact field shall filter database to show only Contacts linked to the selected Account | HIGH |
| FR-204 | Search shall query filtered Contact list by name, email, phone, or Contact ID | HIGH |
| FR-205 | Search results shall display Contact name, email, and role/title | HIGH |
| FR-206 | User can select a Contact from dropdown results | HIGH |
| FR-207 | Selected Contact shall be highlighted and locked in the field | HIGH |
| FR-208 | If Account selection is cleared, Contact field shall be disabled and emptied | HIGH |
| FR-209 | User can clear Contact selection via an "X" button | MEDIUM |
| FR-210 | Contact field is marked as required (*) on the form | HIGH |
| FR-211 | If no Contacts exist for the selected Account, display "No Contacts found" message | MEDIUM |
| FR-212 | Search supports pagination if more than 10 Contacts exist for the Account | MEDIUM |

**Design Details:**
- Field state: Disabled until Account selected, then enabled
- Dependent logic: Account_ID → Filter Contacts
- Minimum characters: 1
- API debounce delay: 300ms
- Max results initially shown: 10
- Field validation: Required (red asterisk)
- Visual feedback: Disabled state (grayed out), Loading spinner during search

---

### 2.3 Related Asset Field (Account-Aware Search)

**User Story:**
> As a case creator, after selecting an Account, I want to search for and select Assets associated with that Account so that the case can track product/resource issues.

**Functional Requirements:**

| Requirement ID | Description | Priority |
|---|---|---|
| FR-301 | Related Asset field shall only be enabled after an Account is selected | HIGH |
| FR-302 | Related Asset field shall display placeholder "Search Assets..." when enabled | HIGH |
| FR-303 | When enabled, Asset field shall filter database to show only Assets owned by or linked to the selected Account | HIGH |
| FR-304 | Search shall query filtered Asset list by Asset name, Asset ID, serial number, or type | HIGH |
| FR-305 | Search results shall display Asset name, Asset ID, type, and status (e.g., Active, Inactive) | HIGH |
| FR-306 | User can select an Asset from dropdown results | HIGH |
| FR-307 | Selected Asset shall be highlighted and locked in the field | HIGH |
| FR-308 | If Account selection is cleared, Related Asset field shall be disabled and emptied | HIGH |
| FR-309 | User can clear Asset selection via an "X" button | MEDIUM |
| FR-310 | Related Asset field is optional (not marked with red asterisk) | HIGH |
| FR-311 | If no Assets exist for the selected Account, display "No Assets found" message | MEDIUM |
| FR-312 | Search supports pagination if more than 10 Assets exist for the Account | MEDIUM |
| FR-313 | Asset field shall support multi-select (user can select multiple Assets for a single case) | MEDIUM |

**Design Details:**
- Field state: Disabled until Account selected, then enabled
- Dependent logic: Account_ID → Filter Assets
- Minimum characters: 1
- API debounce delay: 300ms
- Max results initially shown: 10
- Field validation: Optional (no red asterisk)
- Multi-select: Yes (if FR-313 is implemented)
- Visual feedback: Disabled state, Loading spinner, Selected pills/tags

### 2.4 Command Palette / Command Search Integration

**User Story:**
> As a power user, I want to use a Command Palette to quickly search and navigate to Account, Contact, and Asset data without filling the form step-by-step.

**Functional Requirements:**

| Requirement ID | Description | Priority |
|---|---|---|
| FR-401 | System shall provide a Command Palette accessible via keyboard shortcut (Cmd+K / Ctrl+K) or UI button | MEDIUM |
| FR-402 | Command Palette shall have a search input with placeholder "Type a command or search..." | MEDIUM |
| FR-403 | Palette shall support four search modes: "Search Account", "Search Contact", "Search Asset", "Create Case" | MEDIUM |
| FR-404 | User can type query in Palette to search Accounts across the entire database | MEDIUM |
| FR-405 | User can type query in Palette to search Contacts across the entire database (not limited by Account) | MEDIUM |
| FR-406 | User can type query in Palette to search Assets across the entire database (not limited by Account) | MEDIUM |
| FR-407 | Selecting a result from Palette shall auto-populate the corresponding field in the Create Case form | MEDIUM |
| FR-408 | Palette shall show recent searches and suggestions | LOW |
| FR-409 | Palette shall display keyboard shortcuts (e.g., "⌘P" for Profile) | LOW |
| FR-410 | Palette shall support keyboard navigation (arrow keys, Enter, Escape) | MEDIUM |

**Design Details:**
- Accessibility: Keyboard-first
- Trigger: Cmd+K (Mac), Ctrl+K (Windows/Linux)
- Search modes: Account, Contact, Asset
- Integration: Results populate Create Case form fields
- UX pattern: Command palette (similar to VS Code, Linear, etc.)

## 3. Technical Requirements

### 3.1 Database Integration

| Requirement ID | Description | Priority |
|---|---|---|
| TR-101 | System shall query Account table with indexed fields: name, ID, status | HIGH |
| TR-102 | System shall query Contact table with indexed fields: name, email, phone, account_id | HIGH |
| TR-103 | System shall query Asset table with indexed fields: name, ID, account_id, type, status | HIGH |
| TR-104 | Queries shall execute within 500ms for optimal UX | HIGH |
| TR-105 | System shall implement pagination at database level for large result sets | HIGH |
| TR-106 | System shall cache recent searches to reduce database load | MEDIUM |
| TR-107 | Database queries shall respect user permissions and data access controls | HIGH |

### 3.2 API Endpoints

**Suggested Endpoints:**

```
GET /api/accounts/search?query={query}&limit=10&offset=0
GET /api/contacts/search?query={query}&account_id={id}&limit=10&offset=0
GET /api/assets/search?query={query}&account_id={id}&limit=10&offset=0
GET /api/search?query={query}&type=account|contact|asset&limit=10&offset=0
```

### 3.3 Frontend Implementation

| Requirement ID | Description | Priority |
|---|---|---|
| TR-201 | Implement debounced search to limit API calls (300ms) | HIGH |
| TR-202 | Display loading spinners during API requests | HIGH |
| TR-203 | Implement error handling with user-friendly error messages | HIGH |
| TR-204 | Implement form state management to track dependencies (Account → Contact/Asset) | HIGH |
| TR-205 | Highlight selected values with visual indicators | HIGH |
| TR-206 | Implement accessible autocomplete components (ARIA attributes) | MEDIUM |

## 6. Validation Rules

### 6.1 Field-Level Validation

| Field | Rules | Error Message |
|---|---|---|
| Account | Required, must exist in DB | "Please select a valid Account" |
| Contact | Required, must belong to selected Account | "Please select a Contact for this Account" |
| Related Asset | Optional, must belong to selected Account if provided | "Asset must be associated with the selected Account" |

### 6.2 Form-Level Validation

| Scenario | Action | Message |
|---|---|---|
| User selects Account, then clears it | Clear Contact and Asset fields | No message (automatic) |
| User tries to submit without Account | Block submission | "Account is required" |
| User tries to submit without Contact | Block submission | "Contact is required" |
| User selects Contact from different Account (edge case) | Prevent selection | "This Contact is not associated with the selected Account" |

## 7. Error Handling & Edge Cases

### 7.1 Search Errors

| Scenario | Error Message | Action |
|---|---|---|
| API timeout | "Search timed out. Please try again." | Retry button provided |
| No results found | "No [Accounts/Contacts/Assets] found." | Suggest clearing filter or trying different search term |
| Unauthorized access | "You don't have permission to view this data." | Disable field, contact admin |
| Network error | "Network error. Please check your connection." | Offline mode or retry |

### 7.2 Edge Cases

- **User has no Contacts under Account**: Show "No Contacts found. Create a new Contact?" with link
- **User has no Assets under Account**: Show "No Assets found. Create a new Asset?" with link
- **Account is archived/inactive**: Show warning "This Account is inactive. Continue?" with confirmation
- **Contact moved to different Account after selection**: Validation error on submit with re-selection prompt
- **Concurrent edits**: Show conflict resolution dialog if another user modifies the Account while searching

## 12. Acceptance Criteria

**The feature is complete when:**

✅ User can search and select an Account with autocomplete  
✅ Contact field is disabled until Account is selected  
✅ User can search Contacts filtered by selected Account  
✅ User can search Assets filtered by selected Account  
✅ Clearing Account selection clears Contact and Asset fields  
✅ Command Palette can search all three entities  
✅ Selected values are highlighted and can be cleared via "X" button  
✅ All form fields validate correctly  
✅ API responses are filtered by user permissions  
✅ Keyboard navigation works (Tab, Arrow keys, Enter, Escape)  
✅ Mobile-responsive design works  
✅ Form submits successfully with selected Account, Contact, and Asset IDs  
✅ Error handling displays user-friendly messages  
✅ Performance metrics are met (< 500ms search response)

## 13. Dependencies & Assumptions

### 13.1 Dependencies
- Account, Contact, Asset database tables with proper indexing
- RESTful API with search endpoints
- Form state management library (Redux, Zustand, etc.)
- Autocomplete component library (e.g., Headless UI, Reach UI)

### 13.2 Assumptions
- Database tables exist with proper relationships (Account → Contact, Account → Asset)
- API endpoints return JSON with consistent schema
- User authentication is already implemented
- Role-based access control (RBAC) is available for filtering
- Contact and Asset tables have foreign keys linking to Account

## 14. Glossary

| Term | Definition |
|---|---|
| Account | Organization or customer entity |
| Contact | Person associated with an Account |
| Asset | Product, resource, or item owned by an Account |

| Dependent Field | Field that only functions when another field is populated |
