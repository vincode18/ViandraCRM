# UT Service Console - Functional Requirements Document (FRD)
## Case Module - Three-Column Detail View

**Document Version:** 1.0  
**Created:** 2026-05-28  
**Status:** Draft  
**Audience:** Development Team, QA, Product Managers  
**Module Focus:** Case Management - Detail View Layout

---

## 1. EXECUTIVE SUMMARY

This FRD defines the functional requirements for the UT Service Console Case Module, specifically focusing on the three-column detail view layout that displays when a Case ID is selected. The system will provide users with comprehensive case information in the center column while displaying related linked object data in the left and right columns.

### Key Requirements:
- Three-column layout for case detail view
- Dynamic sidebar navigation updates
- Related object linking (Work Orders, Assets, Fleet Units, etc.)
- Seamless navigation between cases and case list
- Tab-based content organization
- Real-time data synchronization

---

## 2. SCOPE

### 2.1 In Scope
- Case detail view (3-column layout)
- Case information display and editing
- Related object linking and display
- Sidebar navigation updates
- Tab navigation (Details, Feed, Related, Log)
- Breadcrumb navigation
- SLA and compliance tracking
- Timeline and activity feed
- Performance metrics display
- Case status workflow management

### 2.2 Out of Scope
- Case creation (covered separately in Create Case Form)
- Work order management (separate module)
- Asset management (separate module)
- User administration
- System configuration
- Report generation

### 2.3 Dependencies
- Authentication system (user roles and permissions)
- Case management API
- Related object APIs (Work Orders, Assets, Fleet Units, Accounts)
- Real-time notification system
- File storage system (for attachments)
- Audit logging system

## 3. USER STORIES

### 3.1 Primary User Stories

#### US-001: View Case Details in Three-Column Layout
```
As a Service Advisor
I want to view all case details in a three-column layout
So that I can see the full context of the case with related data at a glance

Acceptance Criteria:
1. When I click on a Case ID, the detail view loads
2. The center column displays case information
3. The left column displays related work orders and assets
4. The right column displays SLA tracking and timeline
5. Load time is less than 1.5 seconds
6. All columns are responsive and adjust to screen size
7. Data is automatically refreshed every 30 seconds
8. The view is accessible via keyboard navigation
```

#### US-002: Navigate Back to Case List from Detail View
```
As a Service Advisor
I want to navigate back to the case list from the detail view
So that I can efficiently manage multiple cases

Acceptance Criteria:
1. The sidebar shows "Dashboard" > "All Cases" breadcrumb
2. Clicking "All Cases" returns to the case list
3. The previous list state is preserved (filters, sorting, scroll position)
4. The detail view closes with smooth animation
5. A back button is available in the header
```

#### US-003: Update Sidebar Navigation Based on Current Module
```
As a System User
I want the sidebar to automatically update when viewing cases
So that navigation options are always contextual and relevant

Acceptance Criteria:
1. When viewing case list, sidebar shows case-related options
2. When viewing case detail, sidebar updates to show case-specific filters
3. Sidebar options include: All Cases, Open, In Progress, Pending Review, Closed, Archived
4. Sidebar shows quick links: My Open Cases, SLA Breached, Escalated
5. Sidebar collapses/expands without losing context
6. Active navigation item is highlighted
```

#### US-004: View Related Linked Objects (Left Panel)
```
As a Service Advisor
I want to see all work orders and assets related to a case
So that I can understand the complete scope of work

Acceptance Criteria:
1. Left panel displays Fleet Units associated with the case
2. Fleet units show: ID, Model, Status, Hours, Location
3. Work Orders panel shows: WO Number, Status, Tech assigned, Due date
4. Parts Requests panel shows: Part ID, Description, Qty, Status
5. Account Context shows: Account name, Tier, Primary contact, Contact methods
6. Each section is collapsible/expandable
7. Clicking on linked items navigates to their detail views
8. Related object counts are displayed (e.g., "Fleet Units (4)")
```

#### US-005: View SLA Tracking and Performance Metrics (Right Panel)
```
As a Service Advisor
I want to see SLA status and performance metrics for a case
So that I can prioritize and manage case resolution effectively

Acceptance Criteria:
1. Right panel displays SLA status card
2. SLA card shows: Breached status, Days overdue, Compliance score
3. Performance Metrics table shows: Target vs Actual for key SLAs
4. Timeline flow displays all case lifecycle events
5. Past Cases for Asset shows related historical cases
6. All metrics update in real-time
7. SLA breaches trigger visual alerts (red color)
8. Performance metrics show checkmarks for met SLAs, X for failed
```

#### US-006: Navigate Between Related Cases
```
As a Service Advisor
I want to easily navigate between related cases for the same asset
So that I can understand the history and context

Acceptance Criteria:
1. Past Cases section in right panel lists related cases
2. Each case shows: Case number, Subject, Asset reference, Status
3. Clicking a past case loads its detail view
4. The breadcrumb updates to show the new case
5. Navigation history is maintained (back button works)
```

#### US-007: View and Manage Case Tabs (Details, Feed, Related, Log)
```
As a Service Advisor
I want to view different aspects of a case through tabs
So that I can access relevant information based on my current task

Acceptance Criteria:
1. Details tab shows case information, SLA tracking, description
2. Feed tab shows activity stream with updates and comments
3. Related tab shows linked work orders, parts, documents
4. Log tab shows audit trail of all changes
5. Active tab is highlighted with yellow underline
6. Tab switching is smooth with no page reload
7. Scroll position is maintained within each tab
```

#### US-008: Display Case Status Workflow
```
As a Service Advisor
I want to see the current status and workflow of a case
So that I can track progress and understand current stage

Acceptance Criteria:
1. Case header displays current status badge
2. Status workflow shows: Open → Assigned → In Progress → Resolved → Closed
3. Current status is highlighted (completed steps show green checkmark)
4. Workflow can be expanded/collapsed
5. Status colors are consistent with system design
6. Status cannot be edited from detail view (edit form separate)
```

#### US-009: Search and Filter Related Objects
```
As a Service Advisor
I want to search and filter related work orders and assets
So that I can focus on specific related items

Acceptance Criteria:
1. Left panel has search box for work orders
2. Work orders can be filtered by: Status, Technician, Due Date
3. Assets can be filtered by: Status, Location, Type
4. Search is debounced (300ms) to avoid excessive API calls
5. No results message appears when filters return empty
6. Filter state is maintained during session
```

#### US-010: Display Real-Time Case Updates
```
As a Service Advisor
I want to see real-time updates to case data
So that I always have the most current information

Acceptance Criteria:
1. System polls case data every 30 seconds
2. New updates appear without page reload
3. User is notified of critical updates (SLA breach)
4. Last update timestamp is displayed
5. Auto-refresh can be paused/resumed
6. Update animations are subtle and non-intrusive
```

## 4. FUNCTIONAL REQUIREMENTS

### 4.1 Case Detail View - Center Panel

#### 4.1.1 Case Header Section

**Location:** Top of center panel  
**Components:**
- Case Number (e.g., "01532785")
- Priority Badge (Critical, High, Medium, Low)
- Status Badge (Open, Assigned, In Progress, Resolved, Closed)
- Edit Button
- Change Owner Button
- More Actions Menu (print, export, etc.)

**Requirements:**
1. Case number is displayed in large, bold font (24px)
2. Priority and status badges show appropriate colors:
   - Critical: Red (#ef4444)
   - High: Orange (#f97316)
   - Medium: Yellow (#eab308)
   - Low: Gray (#6b7280)
3. Status badges use corresponding status colors
4. Edit button opens case edit modal or form
5. Change Owner dropdown shows list of available assignees
6. More Actions menu includes: Print, Export as PDF, Share, Delete Draft
7. All actions are permission-based (admin-only for delete, etc.)
8. Header is sticky when scrolling panel content

**HTML/React Structure:**
```jsx
<div className="case-header">
  <div className="header-top">
    <h1 className="case-number">{case.caseNumber}</h1>
    <div className="header-badges">
      <Badge 
        color={getPriorityColor(case.priority)}
        label={case.priority}
      />
      <Badge 
        color={getStatusColor(case.status)}
        label={case.status}
      />
    </div>
    <div className="header-actions">
      <Button 
        icon="edit"
        content="Edit"
        onClick={handleEdit}
      />
      <Button 
        icon="user"
        content="Change Owner"
        onClick={handleChangeOwner}
      />
      <Dropdown icon="ellipsis vertical">
        <Dropdown.Menu>
          <Dropdown.Item 
            icon="print"
            text="Print"
            onClick={handlePrint}
          />
          <Dropdown.Item 
            icon="download"
            text="Export as PDF"
            onClick={handleExportPDF}
          />
          <Dropdown.Item 
            icon="share"
            text="Share"
            onClick={handleShare}
          />
        </Dropdown.Menu>
      </Dropdown>
    </div>
  </div>
  
  {/* Status Workflow */}
  <StatusWorkflow 
    steps={[
      { key: 'open', label: 'Open', completed: true },
      { key: 'assigned', label: 'Assigned', completed: true },
      { key: 'inprogress', label: 'In Progress', completed: true },
      { key: 'resolved', label: 'Resolved', completed: true },
      { key: 'closed', label: 'Closed', completed: true, current: true }
    ]}
  />
</div>
```

#### 4.1.3 Details Tab Content

**Sections:**

##### A. Completion & SLA Tracking

**Required Fields:**
- Start Date OTF Mechanic (editable, with calendar picker)
- Target Date OTF Mechanic (editable, with calendar picker)
- Start Date SLA (editable, with calendar picker)
- Target Date SLA (editable, with calendar picker)
- Backdate Validation section with:
  - Mechanic Start date + time
  - Job Complete date + time
  - Reason for backdating (text field)

**Display Logic:**
1. Dates are displayed in DD/MM/YYYY, HH:MM format
2. Passed dates are shown in red
3. Upcoming dates are shown in green
4. Each date field has an edit icon (pencil)
5. Clicking edit opens an inline date/time picker
6. Changes are auto-saved with confirmation toast
7. Failed validations show error message

**Validation Rules:**
1. Start date cannot be after target date
2. Start date cannot be in the future
3. Target date must be after start date
4. Dates cannot be changed if case is closed (read-only)
5. Only authorized users can edit dates

**Requirements:**
```
- Completion & SLA Tracking card displays all tracking information
- Start Date OTF Mechanic: 26/05/2026, 07:31 [Edit button]
- Target Date OTF Mechanic: 26/05/2026, 11:31 [Edit button]
- Start Date SLA: 26/05/2026, 07:31 [Edit button]
- Target Date SLA: 02/06/2026, 07:31 [Edit button]
- Backdate Validation section shows:
  - Mechanic Start: 26/05/2026, 07:31 ✓
  - Job Complete: 26/05/2026, 15:30 ☐
  - Reason: "Manual Backdate. Initial dispatch delayed due to site access clearance."
- Each field is independently editable
- All changes are tracked in audit log
```

**HTML/React Structure:**
```jsx
<Card className="sla-tracking-card">
  <Card.Content>
    <Card.Header>Completion & SLA Tracking</Card.Header>
  </Card.Content>
  <Card.Content>
    <div className="tracking-field">
      <label>Start Date OTF Mechanic</label>
      <div className="field-value">
        <span>{formatDateTime(case.startDateOTF)}</span>
        <Button 
          icon="edit"
          size="mini"
          onClick={() => setEditingField('startDateOTF')}
        />
      </div>
      {editingField === 'startDateOTF' && (
        <DateTimeInput 
          value={case.startDateOTF}
          onChange={(value) => handleFieldUpdate('startDateOTF', value)}
          onBlur={() => setEditingField(null)}
        />
      )}
    </div>
    
    {/* Repeat for other date fields */}
    
    <div className="backdate-validation">
      <h4>Backdate Validation</h4>
      <div className="backdate-item">
        <span>Mechanic Start</span>
        <span>{formatDateTime(case.mechanicStart)}</span>
        <Icon name="check" color="green" />
      </div>
      <div className="backdate-item">
        <span>Job Complete</span>
        <span>{formatDateTime(case.jobComplete)}</span>
        <Checkbox />
      </div>
      <div className="backdate-reason">
        <label>Reason</label>
        <TextArea 
          value={case.backdateReason}
          onChange={(e) => handleFieldUpdate('backdateReason', e.target.value)}
          placeholder="Provide reason for backdating..."
          rows={3}
        />
      </div>
    </div>
  </Card.Content>
</Card>
```

### 4.3 Case Detail View - Right Panel

#### 4.3.1 SLA Status Card

**Display Requirements:**
1. Large visual alert if SLA is breached
2. Icon: Warning triangle for breach
3. Status: "OVER SLA - BREACHED" (red text)
4. Days overdue (large number)
5. SLA metrics:
   - OTF Mechanic: FAILED/PASSED (with +/- days)
   - OTF Solution: PENDING/PASSED/FAILED
   - SLA Compliance Score (percentage with color coding)
6. "Acknowledge Breach" button (if breached and not acknowledged)

**Color Coding:**
- Breached: Red (#ef4444)
- At Risk: Yellow (#eab308)
- On Track: Green (#22c55e)
- Pending: Gray (#6b7280)

**HTML/React Structure:**
```jsx
<Card className="sla-status-card">
  <Card.Content>
    {case.slaStatus === 'breached' ? (
```

## 7. TECHNICAL REQUIREMENTS

### 7.1 Performance Requirements

- Case detail load time: < 1.5 seconds
- Tab switching: < 200ms
- Navigation between cases: < 500ms
- Search/filter response: < 300ms
- Real-time updates: every 30 seconds (configurable)
- Image/attachment load: lazy load on demand

### 7.2 API Endpoints Required

```
GET /api/cases/{caseId}
  - Returns full case details
  - Includes nested related objects
  - Authorization: Case viewer role

GET /api/cases/{caseId}/related
  - Returns fleet units, work orders, parts, account
  - Authorization: Case viewer role

GET /api/cases/{caseId}/timeline
  - Returns timeline events
  - Pagination: 20 items per request
  - Authorization: Case viewer role

GET /api/cases/{caseId}/feed
  - Returns activity feed
  - Pagination: 20 items per request
  - Real-time updates via WebSocket
  - Authorization: Case viewer role

PUT /api/cases/{caseId}
  - Updates case field
  - Field validation
  - Audit logging
  - Authorization: Case editor role

POST /api/cases/{caseId}/acknowledge-breach
  - Marks SLA breach as acknowledged
  - Timestamp recorded
  - Authorization: Case editor role

GET /api/assets/{assetId}/past-cases
  - Returns related cases for asset
  - Limit: 10 most recent
  - Authorization: Case viewer role
```

**Global State:**
```
- currentCase (object)
- caseLoading (boolean)
- caseError (string)
- activeTab (string: 'details' | 'feed' | 'related' | 'log')
- sidebarNavigation (array)
- breadcrumbs (array)
```

**Cache Strategy:**
```
- Case details: Cache for 5 minutes or until manual refresh
- Timeline events: Cache for 2 minutes
```

## 13. BUSINESS RULES

### 13.1 Case Status Workflow

```
Open (initial)
  ↓
Assigned (when owner assigned)
  ↓
In Progress (when work starts)
  ↓ ↖︎ (can return to if more work needed)
Resolved (when fix complete)
  ↓
Closed (final state)
```

### 13.2 SLA Calculation Rules

**1st Response SLA:** 4 hours from case creation
**Resolution SLA:** 5 days from case creation

**Status:**
- FAILED: If current time > Target date
- PENDING: If current time < Target date and not completed
- PASSED: If completed before target date

**Compliance Score:**
- Calculated as: (Passed SLAs / Total SLAs) × 100
- Updated every 30 seconds
- Breached if any SLA = FAILED

### 13.3 Asset-Case Relationship

- Case must be linked to at least one asset
- Asset can have multiple related cases
- Fleet units are grouped under primary asset
- Related cases shown for primary asset only
