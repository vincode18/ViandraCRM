# Functional Requirements Document (FRD)
## Dynamic Case Details Page — Full Layout & Field Specification

**Document Version:** 3.0
**Last Updated:** June 2026
**Status:** Draft
**Module:** Service Console — Case Management

---

## 1. Purpose

When a user submits the **New Case Form**, the system creates a Case record and renders a **Case Details** page whose **center section** mirrors the exact sections and fields of the New Case Form. Every field is populated from the form input (or left blank if not entered), saved to the database, and **independently editable inline**. The page is a live, dynamic record — not a static snapshot.

The page is organized into three zones:

- **Left panel** — Customer personal data (Account, Contact) and mini Work Order data.
- **Center** — The 9 case sections (the editable record).
- **Right panel** — Related objects: Approval History, Part Request, Work Order, Service Product, Case History.

---

## 2. Core Principles

| ID | Principle |
|----|-----------|
| P-1 | The Case Details center layout = New Case Form layout (same sections, same fields, same order). |
| P-2 | Each field is populated directly from form input. Fields not filled in the form display blank (em-dash `—`), never a default value. |
| P-3 | Each field is **individually editable inline** (pencil icon → edit → save to DB). |
| P-4 | Every save writes to the database immediately and updates `last_modified` + `last_modified_by`. |
| P-5 | The milestone/status starts at **Open** on creation and only advances through user action — it is never auto-set to Closed. |
| P-6 | SLA, OTIF, Billing, and related data adjust dynamically based on the actual field values. |
| P-7 | Required fields (Case Type, Call Type) must be present before the case can progress past Open. |

## 7. Dynamic Data Flow

```
NEW CASE FORM (sections 1–9, partially filled)
        │  submit
        ▼
CREATE CASE RECORD
  - map each form field 1:1
  - empty fields → NULL (shown as "—")
  - status = Open, milestone = Open
  - Created By, SF Case Number set
        │
        ▼
RENDER CASE DETAILS (same section layout)
  - center: 9 sections populated
  - left: Account/Contact/Mini WO
  - right: related objects
        │
        ▼
INLINE EDIT (any field)
  - pencil → editor → save
  - write to DB
  - update Last Modified / Last Modified By
  - append to Case History (right panel)
  - re-run dependent logic (SLA, OTIF, Billing auto-fill, milestone)
```

**Field-level save contract:**
1. User edits one field and confirms.
2. `PATCH /api/cases/{id}` with `{ field, value }`.
3. DB updated; `last_modified` + `last_modified_by` refreshed.
4. Response returns the canonical value; UI re-renders that field only.
5. History entry created. Dependent fields recalculated (e.g., changing Target Date OTIF → updates OTIF Status; changing Account → refreshes Billing auto-fill and Left panel).

## 8. Status & Milestone Logic

- Milestone bar: **Open → Assigned → In Progress → Resolved → Closed**.
- Created case begins at **Open**; the bar shows Open active and the rest inactive.
- Each stage sets its corresponding timestamp in Section 8 (Assigned/In Progress/Resolved).
- **Closed** is reached only by explicit user action ("Mark Status as Complete" / milestone select) and requires a Completion Note and Close Reason.
- Status field in Section 3 stays in sync with the milestone bar.
- The milestone must **not** auto-close on creation or on resolve.

---

## 9. Empty Field Handling

| Rule | Behavior |
|------|----------|
| Not filled in form | Store NULL, render `—`, field still visible & editable |
| No defaults | Never substitute config/default values for blank fields |
| Required missing | Block progression past Open; highlight Case Type / Call Type |
| Conditional required | Reason Backdate, NOT-OTIF reasons, Close Reason enforced only when their trigger condition is met |

## 10. Validation Rules

| ID | Rule | Message |
|----|------|---------|
| V-1 | Account & Contact required at creation | "Account and Contact are required" |
| V-2 | Case Type & Call Type required to leave Open | "Case Type and Call Type are required" |
| V-3 | Contact must belong to Account | "Contact not associated with selected Account" |
| V-4 | Backdate ≥ Case Start unless reason given | "Provide a reason for backdating" |
| V-5 | Close requires Completion Note + Close Reason | "Completion Note and Close Reason required to close" |
| V-6 | Email format valid (Informant / Billing Contact) | "Enter a valid email" |
| V-7 | Target dates ≥ Start dates (OTIF) | "Target date cannot precede start date" |

## 11. API Sketch

```
POST   /api/cases                 → create case from form (all 9 sections)
GET    /api/cases/{id}            → full case (center + left + right data)
PATCH  /api/cases/{id}            → update one or more fields (inline edit)
POST   /api/cases/{id}/close      → close (requires completion note + close reason)
GET    /api/cases/{id}/history    → Case History feed (right panel)
GET    /api/cases/{id}/related    → approvals, part requests, work orders, service products
```

`PATCH` body example:
```json
{ "section": "case_details", "field": "case_type", "value": "Breakdown" }
```

Response includes recalculated dependent fields:
```json
{
  "field": "case_type",
  "value": "Breakdown",
  "last_modified": "2026-06-01T05:45:00Z",
  "last_modified_by": "System Administrator",
  "recalculated": { "sla_resolution_target": "2026-06-02T07:31:00Z" }
}
```

## 12. Data Model (Indicative)

```sql
CREATE TABLE cases (
  case_id            VARCHAR PRIMARY KEY,
  sf_case_number     VARCHAR,
  -- Section 1
  account_id         VARCHAR NOT NULL,
  contact_id         VARCHAR NOT NULL,
  asset_id           VARCHAR,
  plant              VARCHAR,
  service_area       VARCHAR,
  -- Section 2
  informant_name     VARCHAR,
  informant_email    VARCHAR,
  informant_position ENUM('ASC','Sales','BC'),
  -- Section 3
  subject            VARCHAR,
  parent_case_id     VARCHAR,
  case_description   TEXT,
  direction          VARCHAR,
  location           VARCHAR,
  case_origin        VARCHAR,
  smr_problem        VARCHAR,
  datetime_opened    TIMESTAMP,
  object_part        VARCHAR,
  specific_object_part VARCHAR,
  case_type          VARCHAR NOT NULL,
  cause              VARCHAR,
  call_type          VARCHAR NOT NULL,
  damage             VARCHAR,
  subcall_type       VARCHAR,
  status             ENUM('Open','Assigned','In_Progress','Resolved','Closed','Cancelled'),
  description_update TEXT,
  sap_status         VARCHAR,
  cs_rating          VARCHAR,
  close_reason       VARCHAR,
  emr                VARCHAR,
  wa_update_progress TEXT,
  trouble_date       DATE,
  wa_closing_update  TEXT,
  wa_number          VARCHAR,
  wa_description     TEXT,
  -- Section 4
  otif_mech_start    TIMESTAMP,
  otif_mech_target   TIMESTAMP,
  otif_sol_start     TIMESTAMP,
  otif_sol_target    TIMESTAMP,
  -- Section 5
  backdate_mech      TIMESTAMP,
  backdate_sol       TIMESTAMP,
  reason_backdate    VARCHAR,
  reason_others      TEXT,
  -- Section 6
  otif_mech_status   VARCHAR,
  not_otif_mech      BOOLEAN,
  not_otif_mech_reason VARCHAR,
  om_compensation    VARCHAR,
  otif_sol_status    VARCHAR,
  not_otif_sol       BOOLEAN,
  not_otif_sol_reason VARCHAR,
  -- Section 7
  billing_account    VARCHAR,
  billing_sales_office VARCHAR,
  billing_division   VARCHAR,
  billing_sales_office_code VARCHAR,
  billing_contact_name VARCHAR,
  -- Section 8
  dt_assigned        TIMESTAMP,
  approval_status    VARCHAR,
  dt_in_progress     TIMESTAMP,
  need_man_power     BOOLEAN,
  dt_resolved        TIMESTAMP,
  case_cancel        BOOLEAN,
  dt_supervisor_approve TIMESTAMP,
  -- Section 9 / system
  milestone_status   ENUM('Open','Closed') DEFAULT 'Open',
  created_by         VARCHAR,
  created_at         TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  last_modified      TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  last_modified_by   VARCHAR
);

CREATE TABLE case_history (
  id           BIGINT PRIMARY KEY AUTO_INCREMENT,
  case_id      VARCHAR,
  field        VARCHAR,
  old_value    TEXT,
  new_value    TEXT,
  changed_by   VARCHAR,
  changed_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 13. Acceptance Criteria

✅ Case Details center renders all 9 sections in the same order/layout as the New Case Form.
✅ Every listed field is present and populated from form input, or shows `—` if blank.
✅ Each field is individually editable inline and persists to the database on save.
✅ `Last Modified` / `Last Modified By` update on every edit; Case History logs the change.
✅ Left panel shows Account, Contact, and Mini Work Order, updating with Section 1 changes.
✅ Right panel shows Approval History, Part Request, Work Order, Service Product, Case History.
✅ Milestone starts at Open and only closes on explicit user action with Completion Note + Close Reason.
✅ Required fields (Case Type, Call Type) block progression past Open until filled.
✅ OTIF/SLA/Billing values recalculate dynamically from their source fields.
✅ Conditional fields (backdate reasons, NOT-OTIF reasons) enforce only when triggered.
