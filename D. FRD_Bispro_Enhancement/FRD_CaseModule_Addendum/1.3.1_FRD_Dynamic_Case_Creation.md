# Functional Requirements Document (FRD)
## Dynamic Case Creation with Real-time Data Synchronization and Business Logic

**Document Version:** 2.0  
**Last Updated:** June 2026  
**Status:** Draft  
**Type:** Case Management System Enhancement

---

## 1. Executive Summary

This document outlines the functional requirements for a **dynamic case creation system** where newly created cases reflect the exact data entered in the Create Case Form without static pre-processing. The system implements real-time business logic that automatically calculates SLA, assigns assets, determines priority, and manages case status based on form inputs. All form fields that are not filled remain blank, and the case details page mirrors the form submission state with live updates.

---

## 2. Problem Statement

**Current State Issues:**
- Cases created show pre-processed data that doesn't match form inputs
- SLA, asset assignments, and priority calculations happen before user confirmation
- Incomplete fields show default values instead of remaining blank
- Case details page shows "Closed" status prematurely
- No real-time synchronization between form and case details
- Business logic is static and doesn't adapt to actual form input

**Desired State:**
- Case details immediately reflect form submission data
- Empty form fields remain blank in case record
- Business logic executes dynamically based on actual inputs
- SLA, assets, and status adjust in real-time as case is processed
- Case lifecycle progresses through proper workflow states

### 4.2 Dynamic Business Process Logic

**Requirement ID: BR-201**  
**Title:** Real-time Business Logic Execution  
**Priority:** HIGH

The system shall implement a **dynamic business logic engine** that processes case data in real-time:

#### 4.2.1 SLA Calculation Logic

**Requirement ID: BR-201-A**

| Trigger | Logic | Output |
|---------|-------|--------|
| Case Created | Calculate SLA based on Priority + Category | SLA Target Time |
| Priority Changed | Recalculate SLA immediately | New SLA Target |
| Category Updated | Adjust SLA multiplier based on category | Revised SLA |
| Account Changed | Apply account-specific SLA rules | Custom SLA |

**SLA Matrix:**

```
Priority × Category = Response Time

PRIORITY LEVELS:
- Critical: 1 hour response, 4 hours resolution
- High: 4 hours response, 8 hours resolution
- Medium: 8 hours response, 24 hours resolution
- Low: 24 hours response, 5 days resolution

CATEGORY MULTIPLIERS:
- Technical: 1.0x (standard)
- Billing: 1.2x (longer SLA)
- General: 1.0x (standard)
- Emergency: 0.5x (expedited)

ACCOUNT MODIFIERS:
- Enterprise Tier: 0.8x (faster SLA)
- Standard Tier: 1.0x (standard SLA)
- Basic Tier: 1.2x (longer SLA)
```

**Example Calculation:**
```
Input: Priority=High, Category=Technical, Account=Enterprise

SLA Calculation:
1. Base SLA (High Priority): 4 hours response
2. Category Multiplier (Technical): 1.0x
3. Account Modifier (Enterprise): 0.8x
4. Final SLA: 4 × 1.0 × 0.8 = 3.2 hours response time

Output:
- Response Target: Current_Time + 3h 12min
- Resolution Target: Current_Time + 6h 24min
- Status: SLA Active (Open)
```

**Implementation:**
- SLA calculations trigger when case is created
- Recalculate in real-time if Priority, Category, or Account changes
- Display SLA countdown on case details page
- Update SLA status: On Track / At Risk / Breached

#### 4.2.2 Asset Assignment Logic

**Requirement ID: BR-201-B**

When an Asset is selected in the Create Case Form:

| Action | Logic | Output |
|--------|-------|--------|
| Asset Selected | Link asset to case immediately | Asset ID stored |
| Asset Status | Check asset current status | Display status badge |
| Asset History | Show past cases for this asset | Related cases list |
| Asset Condition | Trigger maintenance if needed | Alert generated |
| Multiple Assets | Support multi-asset case | All assets linked |

**Asset Assignment Rules:**
```
IF Asset is selected in form THEN
  - Link asset to case (1:many relationship)
  - Create asset-case relationship record
  - Retrieve asset current status (Active/Inactive/In-Service)
  - Display status on case details
  - Query past cases for this asset (max 5 recent)
  - Check asset maintenance schedule
  - If maintenance due → Create maintenance alert
  - Update asset "last case date" field
ENDIF

IF Multiple Assets selected THEN
  - Store all asset IDs in case.asset_ids (array)
  - Display all assets in "Related Assets" section
  - Calculate combined SLA if assets have different SLA tiers
ENDIF
```

---

#### 4.2.3 Contact-Account Relationship Logic

**Requirement ID: BR-201-C**

When Contact and Account are selected:

```
IF Account selected AND Contact selected THEN
  - Verify Contact belongs to selected Account
  - Retrieve Contact role/title
  - Retrieve Contact communication preferences
  - Pre-fill Contact details (email, phone)
  - Link Primary Contact to case
  - Query other Contacts in Account (for escalation)
  
IF Account changed THEN
  - Validate Contact still belongs to new Account
  - IF Contact no longer belongs → Clear Contact field
  - Display warning: "Contact not in new Account"
  - Enable user to select new Contact
ENDIF

IF Contact changed THEN
  - Update Contact info on case details
  - Refresh Contact history for this Contact
  - Update communication method based on preferences
ENDIF
```

### 4.4 Case Status & Milestone Lifecycle

**Requirement ID: MS-401**  
**Title:** Dynamic Milestone Management  
**Priority:** HIGH

Cases progress through defined status states dynamically:

#### 4.4.1 Case Status States

```
┌─────────────────────────────────────────────────────┐
│              CASE LIFECYCLE FLOW                     │
└─────────────────────────────────────────────────────┘

                    DRAFT/OPEN
                        │
                        ▼
                    IN-PROGRESS
                    (When assigned)
                        │
          ┌─────────────┼─────────────┐
          ▼             ▼             ▼
      WAITING     ON-HOLD      ESCALATED
      (Customer)  (Temporary)  (To team)
          │             │             │
          └─────────────┼─────────────┘
                        ▼
                    IN-PROGRESS
                    (Work continues)
                        │
          ┌─────────────┴─────────────┐
          ▼                           ▼
      RESOLVED                    CANCELLED
      (Solution provided)         (Not needed)
          │                           │
          ▼                           ▼
      CLOSED                      CLOSED
   (Final state)              (Final state)
```

**Status Definitions:**

| Status | Description | SLA | Milestone | Auto-trigger |
|--------|-------------|-----|-----------|--------------|
| DRAFT | Form submitted, not yet activated | Not started | Open | On form submit |
| OPEN | Case created, awaiting assignment | Active | Open | Case creation |
| IN-PROGRESS | Assigned to owner, work started | Active | Open | On assignment |
| WAITING | Awaiting customer response | Paused | Open | Owner action |
| ON-HOLD | Temporarily paused | Paused | Open | Owner action |
| ESCALATED | Escalated to senior team | Active (faster SLA) | Open | Owner action |
| RESOLVED | Solution provided, awaiting closure | Completed | Open → Closed | Owner action |
| CLOSED | Case completed | Stopped | Closed | Auto (post-resolution) |
| CANCELLED | Case cancelled | Stopped | Closed | Owner action |

**Key Requirement:**
- **Milestone shall remain OPEN until explicitly closed**
- Milestone should NOT auto-close when case is resolved
- Only explicit "Close Case" or "Mark as Complete" action closes milestone
- Completion notes are required to close milestone

### 4.5 Empty Field Handling

**Requirement ID: EFH-501**  
**Title:** NULL/Blank Field Management  
**Priority:** HIGH

When form fields are not filled:

**Requirement ID: EFH-501-A**

```
EMPTY FIELD RULES:

┌─────────────────────────────────────────────────────┐
│ FIELD NOT FILLED IN FORM                            │
├─────────────────────────────────────────────────────┤
│                                                      │
│ DO NOT:                        DO:                 │
│ ❌ Apply default value        ✅ Store as NULL    │
│ ❌ Show placeholder           ✅ Display blank     │
│ ❌ Pre-populate from config   ✅ Show empty state  │
│ ❌ Hide the field             ✅ Keep field visible│
│ ❌ Skip validation            ✅ Mark optional     │
│                                                      │
└─────────────────────────────────────────────────────┘

IMPLEMENTATION:

ON FORM SUBMIT:
  FOR each field in form:
    IF field.value === "" OR field.value === null THEN
      case.field = NULL
    ELSE
      case.field = field.value
    END IF
  END FOR

ON CASE DETAILS DISPLAY:
  FOR each field in case record:
    IF field === NULL THEN
      display_field = "" (blank)
      field_ui_state = "not-filled"
    ELSE
      display_field = field_value
      field_ui_state = "filled"
    END IF
  END FOR

OPTIONAL FIELDS (Can be blank):
- Category
- Sub-Category
- Related Asset (unless required by policy)
- Description (if not required)
- Custom fields

REQUIRED FIELDS (Must be filled):
- Account (marked with *)
- Contact (marked with *)
- Case Type (if required)
- Priority (if required)
```

### 4.7 Business Logic Execution Order

**Requirement ID: BLO-701**  
**Title:** Sequential Business Logic Processing  
**Priority:** HIGH

When case is created, business logic executes in this order:

```
CASE CREATION BUSINESS LOGIC SEQUENCE:

Step 1: VALIDATE FORM
├─ Check required fields (Account, Contact)
├─ Validate field formats
├─ Check Account-Contact relationship
└─ Status: VALIDATION PENDING

Step 2: CREATE CASE RECORD
├─ Generate Case ID
├─ Store all form data (blank fields as NULL)
├─ Record creation timestamp
├─ Set status = DRAFT/OPEN
└─ Status: CASE CREATED

Step 3: CALCULATE SLA
├─ Retrieve Priority from form
├─ Retrieve Category from form
├─ Retrieve Account tier
├─ Apply SLA matrix rules
├─ Calculate Response & Resolution targets
└─ Status: SLA CALCULATED

Step 4: ASSIGN ASSET (if selected)
├─ Verify asset exists
├─ Link asset to case
├─ Retrieve asset status
├─ Query asset history (past 5 cases)
├─ Check maintenance schedule
└─ Status: ASSET ASSIGNED

Step 5: RETRIEVE CONTACT DETAILS
├─ Load contact information
├─ Retrieve contact preferences
├─ Retrieve communication method
├─ Load contact history
└─ Status: CONTACT LOADED

Step 6: SET INITIAL STATUS
├─ Status = OPEN (waiting for assignment)
├─ Milestone = OPEN
├─ Owner = Case Creator (default)
└─ Status: READY FOR PROCESSING

Step 7: INITIALIZE CASE DETAILS PAGE
├─ Display all form data
├─ Show SLA countdown
├─ Show asset information
├─ Show contact information
├─ Show empty fields as blank
└─ Status: READY FOR DISPLAY

Step 8: TRIGGER NOTIFICATIONS
├─ Notify case owner
├─ Notify assigned team (if auto-assigned)
├─ Send contact notification (if configured)
└─ Status: NOTIFICATIONS SENT

TOTAL EXECUTION TIME: < 2 seconds
```

## 5. Data Model & Schema

### 5.1 Case Record Structure

```sql
CREATE TABLE cases (
  case_id VARCHAR PRIMARY KEY,
  account_id VARCHAR NOT NULL,
  contact_id VARCHAR,
  case_type VARCHAR,
  priority VARCHAR,
  category VARCHAR,
  sub_category VARCHAR,
  subject VARCHAR,
  description TEXT,
  asset_ids JSON,  -- Array of asset IDs
  owner_id VARCHAR,
  status ENUM('DRAFT', 'OPEN', 'IN_PROGRESS', 'WAITING', 'ON_HOLD', 'ESCALATED', 'RESOLVED', 'CLOSED', 'CANCELLED'),
  milestone_status ENUM('OPEN', 'CLOSED'),
  
  sla_response_target TIMESTAMP,
  sla_resolution_target TIMESTAMP,
  sla_status ENUM('ON_TRACK', 'AT_RISK', 'BREACHED'),
  
  created_by VARCHAR,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  completed_at TIMESTAMP,
  completion_note TEXT,
  
  -- Soft fields for form state tracking
  form_data_source VARCHAR DEFAULT 'CREATE_FORM',
  is_synced BOOLEAN DEFAULT TRUE,
  last_sync_timestamp TIMESTAMP
);

CREATE TABLE case_assets (
  id INT PRIMARY KEY AUTO_INCREMENT,
  case_id VARCHAR FOREIGN KEY REFERENCES cases(case_id),
  asset_id VARCHAR NOT NULL,
  linked_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE case_timeline (
  id INT PRIMARY KEY AUTO_INCREMENT,
  case_id VARCHAR FOREIGN KEY REFERENCES cases(case_id),
  event_type VARCHAR,  -- 'STATUS_CHANGE', 'NOTE_ADDED', 'PRIORITY_CHANGED', etc.
  old_value VARCHAR,
  new_value VARCHAR,
  changed_by VARCHAR,
  changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  description TEXT
);

CREATE TABLE case_completion_notes (
  id INT PRIMARY KEY AUTO_INCREMENT,
  case_id VARCHAR FOREIGN KEY REFERENCES cases(case_id),
  note TEXT NOT NULL,
  created_by VARCHAR,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 7. Integration Points

### 7.1 Real-time Synchronization (WebSocket)

```javascript
// WebSocket subscription for live case updates
ws.subscribe(`case/${case_id}/updates`, (message) => {
  
  if (message.type === 'FIELD_UPDATED') {
    // Update UI in real-time
    updateCaseField(message.field, message.value);
    highlightChange(message.field);
  }
  
  if (message.type === 'SLA_RECALCULATED') {
    // Update SLA countdown
    updateSLATimer(message.sla);
    updateSLAStatus(message.status);
  }
  
  if (message.type === 'STATUS_CHANGED') {
    // Update milestone and status
    updateCaseStatus(message.status);
    updateMilestoneStatus(message.milestone_status);
  }
  
  if (message.type === 'TIMELINE_EVENT') {
    // Add to timeline
    addTimelineEvent(message.event);
  }
});
```
