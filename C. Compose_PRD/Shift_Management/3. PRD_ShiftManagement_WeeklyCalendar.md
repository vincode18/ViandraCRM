## Shift Management — Weekly Calendar View

## 1. Background & Problem Statement

The current Shift Management screen renders shifts on a **horizontal hourly Gantt chart** (Day / 3-Day / Week tabs). This model was designed for precise time-block scheduling, but in practice, shift planning for Service Resources operates at the **daily granularity** — a technician is assigned to a morning shift, afternoon shift, or full-day shift on a given day, not to an arbitrary two-hour window.

**Pain points observed:**

- The hourly timeline is too narrow on the Week view — shift labels are truncated and unreadable.
- Dispatchers have to horizontally scroll a dense grid to get a weekly overview.
- Comparing a resource's coverage across the week requires scanning a long horizontal bar, which is cognitively taxing.
- The current model discourages "who is free on Thursday?" questions — the most common dispatch query.

---

## 2. Proposed Change

Replace the **Week view tab** with a **Weekly Calendar Grid** (similar to Google Calendar / Salesforce FSL Shifts), while keeping the **Day** and **3-Day** tabs unchanged.

In the weekly view:
- Columns = **days of the week** (Mon → Sun, 7 columns).
- Rows = **each Service Resource**.
- Each cell shows the **shift card(s)** assigned to that resource on that day.
- A shift card shows: Shift ID, Shift Type / Label, Time Range, and a colour band by shift type.

## 4. Non-Goals

- **Not** replacing the Day or 3-Day views.
- **Not** adding time-of-day sub-rows within the weekly calendar (that is the Day view's job).
- **Not** implementing drag-and-drop in this iteration (deferred to PRD-SM-002).


## 8. Data Model

No backend changes required. The existing shift object already contains:

```json
{
  "id": "SFT-3061099",
  "resourceId": "bs-001",
  "date": "2026-05-23",
  "startTime": "08:00",
  "endTime": "17:00",
  "type": "Day Shift",
  "status": "Confirmed"
}
```

The `useShiftWeekData` hook groups shifts into:

```js
{
  "bs-001": {
    "2026-05-23": [shift1],
    "2026-05-25": [shift2, shift3],
  },
  "am-001": { ... }
}
```

## 10b. Shift Template Enhancement (PRD-SM-001a)

### 10b.1 Background

Dispatchers repeatedly assign the *same* recurring shift patterns (e.g. "Day Shift 08:00–17:00", "Night Shift 20:00–06:00") to Service Resources (Mechanics). Re-entering start/end time and type for every cell is slow and error-prone. **Shift Templates** let a dispatcher define a reusable shift pattern once and apply it to a mechanic with a single action.

### 10b.2 Template Data Model

A Shift Template is a lightweight, reusable definition (no resource, no date):

```json
{
  "id": "tpl-day",
  "name": "Day Shift",
  "type": "Day Shift",
  "start": "08:00",
  "end": "17:00"
}
```

Default seed templates: **Day Shift**, **Normal**, **Night Shift**, **On Call**, **Emergency Standby**. Dispatchers may add custom templates (name, type, start, end).

### 10b.3 Three Application Modes

| Mode | Trigger | Behaviour |
|------|---------|-----------|
| **M1 — Quick-apply picker** | Click an empty cell with no active template | A popover lists all templates; selecting one instantly creates that shift for the mechanic + date. |
| **M2 — Template panel + click-apply** | Select a template in the side panel, then click empty cells | The chosen template becomes the "active brush"; each empty-cell click applies it. Click the template again to deselect. (Drag-to-apply deferred to PRD-SM-002.) |
| **M3 — Bulk weekly pattern** | "Apply Pattern" on a resource row → dialog | Pick a template + tick the weekdays (Mon–Sun); one action creates shifts for all selected days that are currently empty. |

### 10b.4 Template Side Panel

- Collapsible right-hand panel titled **Shift Templates**.
- Each template row: colour band (by shift type), name, time range, type label.
- An **active** template is visually highlighted (accent ring) and shown in the header as the active brush.
- **+ Add Template** opens an inline form (name, type dropdown, start, end).

### 10b.5 Rules

- Templates only create shifts in **empty** cells; occupied cells are skipped (no overwrite in this iteration).
- New shifts default to status **Tentative**.
- Generated shift IDs follow the existing `SFT-XXXXXXX` convention.
- Bulk apply reports how many shifts were created via a toast.

### 10b.6 Acceptance Criteria (Template Enhancement)

- [ ] Side panel lists seed templates with correct colour coding.
- [ ] Selecting a template highlights it and enables click-apply on empty cells.
- [ ] Clicking an empty cell with no active template opens the quick-apply picker.
- [ ] Applying a template creates a shift card with the template's type/time and Tentative status.
- [ ] "Apply Pattern" creates shifts across the selected weekdays (empty cells only).
- [ ] Adding a custom template makes it immediately available in all three modes.
