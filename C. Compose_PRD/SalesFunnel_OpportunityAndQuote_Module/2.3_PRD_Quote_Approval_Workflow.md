## 1. Overview

This PRD defines the **approval and negotiation workflow** that governs Quote lifecycle after initial submission. It covers the SDH (Service Delivery Head) approval queue experience, rejection-and-revision flow, version management, and the automated Opportunity price write-back that concludes a successful negotiation.

Where `PRD_Quotation_Module.md` describes what the Quote object is and how it is created, this document describes how it is **reviewed, decided upon, revised, and resolved**.

---

## 2. Problem Statement

Quote approval in field service environments is typically handled through informal channels (email, Slack, verbal sign-off). This creates three failure points:

**Accountability gap** — no record of who approved what price at what time.  
**Revision chaos** — when a quote is rejected, the revised version has no formal link to the original, making it impossible to audit negotiation history.  
**Stale Opportunity data** — even when prices are agreed upon, the Opportunity line items are updated manually and inconsistently.

This workflow module closes all three gaps.

## 4. Actors

| Actor | Role in this Flow |
|-------|------------------|
| **Rep / Field Agent** | Submits initial Quote; receives approval/rejection notifications; creates revised Quotes |
| **SDH (Service Delivery Head)** | Reviews Quotes in approval queue; approves or rejects with comments |
| **Opportunity Owner** | Receives notification when Opportunity prices are updated; may or may not be the same as the Rep |
| **System** | Routes notifications, manages version state, executes Opportunity price write-back |

## 5. Workflow Phases

### Phase 1: Quote Submission (Rep)

The Rep completes Quote line items in Draft state and clicks **"Submit for Approval"**.

**Pre-submission validation:**

| Validation | Error Message |
|------------|---------------|
| No line items added | "Add at least one product or part before submitting." |
| Grand Total is zero | "Quote total cannot be zero. Review your line items." |
| Quote already submitted | "This Quote has already been submitted and is pending approval." |
| Duplicate active pending Quote | "An active Quote is already pending approval for this Opportunity." |

**On successful submission:**
- Quote `Status` → `Submitted`
- `SubmittedAt__c` and `SubmittedBy__c` populated
- Line Items locked (read-only)
- In-app notification dispatched to all users with SDH role
- Quote appears in SDH's Pending Approvals queue

---

### Phase 2: SDH Review

#### 2.1 Pending Approvals Queue

SDH users access a dedicated **Pending Approvals** section in the navigation. The queue displays:

| Column | Notes |
|--------|-------|
| Quote Number + Version | e.g., `Q-2025-001 v2` |
| Opportunity Name | Linked to Opportunity record |
| Account Name | — |
| Submitted By | Rep name |
| Submitted Date | — |
| Grand Total | Formatted currency |
| Age (days pending) | Calculated from submission date; highlights in warning colour at 2+ days |

Queue is sorted by **Submitted Date ascending** (oldest first) by default. SDH can filter by Account or Opportunity.

#### 2.2 Quote Review View

Clicking a queue item opens the full Quote detail in a read-only review panel. The SDH sees:

- Quote header (Number, Version, Account, Opportunity, Contact, Expiration)
- All Line Items: Product, SKU, Qty, Unit Price, Discount, Line Total
- Quote totals: Subtotal, Discount, Tax, Shipping, Grand Total
- Version history panel (prior versions if this is v2+, with their status and rejection reasons)
- Submission context: who submitted, when, any notes

#### 2.3 Approval Decision Controls

Two primary actions are available at the top and bottom of the review panel:

**Approve** — opens a confirmation modal with optional comments field. On confirmation:
- Quote `Status` → `Approved`
- `ApprovedRejectedAt__c` and `ApprovedRejectedBy__c` populated
- Opportunity price write-back triggered (see Phase 3)
- Rep notification dispatched

**Reject** — opens a rejection modal with a **required** rejection reason textarea. On confirmation:
- Quote `Status` → `Rejected`
- `RejectionReason__c` populated
- `ApprovedRejectedAt__c` and `ApprovedRejectedBy__c` populated
- Line Items unlocked on the Rep's side (Draft revision mode)
- Rep notification dispatched with rejection reason

> **Validation:** SDH cannot submit the rejection modal without entering a rejection reason. The confirm button is disabled until the textarea contains at least 10 characters.

---

### Phase 3: Opportunity Price Write-back (System, on Approval)

This phase is fully automated and triggered the moment the SDH confirms approval.

#### 3.1 Write-back Logic

```
FOR EACH approved Quote Line Item:
  FIND matching Opportunity Line Item by Product2Id
  IF match found:
    UPDATE OpportunityLineItem.UnitPrice = QuoteLineItem.UnitPrice
    UPDATE OpportunityLineItem.Quantity  = QuoteLineItem.Quantity
    UPDATE OpportunityLineItem.Discount  = QuoteLineItem.Discount
  IF no match found:
    CREATE new OpportunityLineItem from Quote Line Item
```

After write-back:
- Opportunity `Amount` field recalculates automatically (Salesforce standard rollup)
- Quote `IsSyncing` → `true`
- Opportunity activity log entry created: _"Prices updated from approved Quote [Q-2025-001 v2] by [SDH Name]"_
- Opportunity quote status indicator → `Quote Approved`
- Opportunity Owner receives in-app notification

#### 3.2 Write-back Failure Handling

| Failure Scenario | System Behaviour |
|------------------|----------------|
| Product in Quote not in Opportunity's Pricebook | Log error; notify Admin; write-back pauses; SDH and Rep notified |
| Opportunity is Closed Won/Lost at time of write-back | Block write-back; notify Admin; Quote remains Approved |
| Partial write-back failure (some lines succeed) | Rollback all; treat as full failure; notify Admin |

All write-back failures are logged to an `IntegrationErrorLog__c` custom object for Admin review.

---

### Phase 4: Rejection & Revision (Rep creates Quote v2)

#### 4.1 Rep Notification

On rejection, the Rep receives:
- In-app notification: _"Your Quote [Q-2025-001] has been rejected by [SDH Name]."_
- Notification includes SDH's rejection reason (truncated to 160 chars; full text visible on Quote)

#### 4.2 Creating a Revised Quote

From the Opportunity detail page (or from the rejected Quote record), the Rep sees a **"Submit Revised Quote"** action. This action:

1. Creates a new Quote record with:
   - `Version__c` = previous version + 1
   - `ParentQuoteId__c` = ID of the rejected Quote
   - Status = `Draft`
   - All Line Items copied from the previous version (editable)
   - Header fields copied (Account, Contact, Expiration, Description)

2. Previous Quote's `Status` → `Superseded` (if it was Rejected and a new version is now in Draft; note: Rejected status is preserved until the new version is submitted)

3. The Opportunity Quote indicator → `Revision in Progress`

The Rep can then modify line items and re-submit. The revised Quote follows the identical Phase 1 → 2 → 3/4 flow.

#### 4.3 Version Chain

The system maintains a linked version chain for audit purposes:

```
Q-2025-001 v1  [Rejected]  ──►  Q-2025-001 v2  [Rejected]  ──►  Q-2025-001 v3  [Approved]
     │                                │                                │
  Rejection                        Rejection                      Write-back
  Reason: "..."                    Reason: "..."                  to Opportunity
```

All versions are visible in the Quote's **Version History** tab, which is read-only for all parties.

## 6. Approval Delegation (v1 Scope: Limited)

In v1, the SDH approval role is assigned at the user-role level. There is no automated delegation (e.g., if SDH is on leave). Out-of-office delegation is an Admin-level task: the Admin temporarily assigns the SDH role to a deputy.

Auto-delegation (based on OOO calendar or escalation timer) is deferred to v2.

## 7. Notifications Specification

### 7.1 Notification Events

| Event | Recipient(s) | Message |
|-------|-------------|---------|
| Quote submitted | All SDH-role users | "Quote [Q-Number] v[N] from [Rep Name] is awaiting your approval — [Opportunity Name] · [Grand Total]" |
| Quote approved | Rep (submitter) | "Your Quote [Q-Number] v[N] has been approved. Opportunity prices have been updated." |
| Quote rejected | Rep (submitter) | "Your Quote [Q-Number] v[N] has been rejected. Reason: [RejectionReason truncated]" |
| Prices written to Opportunity | Opportunity Owner | "Opportunity [Opp Name] prices updated from approved Quote [Q-Number] v[N]." |
| Write-back failure | Admin users | "Write-back failed for Quote [Q-Number]. Manual intervention required." |

### 7.2 Notification Design

- Notifications are displayed in the console's global notification bell (top nav)
- Unread notifications show a numeric badge
- Clicking a Quote notification deep-links to the Quote detail view
- Notifications are not cleared automatically; the user must mark as read

## 8. Audit Trail

Every state transition in the Quote lifecycle is recorded with:

| Field | Details |
|-------|---------|
| Previous Status | Before the transition |
| New Status | After the transition |
| Actor | User who triggered the transition |
| Timestamp | UTC |
| Notes | Rejection reason, approval comments, system messages |

The audit trail is exposed via the **Quote History** tab on the Quote record and is read-only. It cannot be deleted by any user, including Admins (append-only log).

## 9. Edge Cases & Rules

| # | Scenario | Rule |
|---|----------|------|
| EC-01 | SDH is also the Rep who submitted the Quote | System blocks self-approval; SDH must delegate to another SDH user |
| EC-02 | Rep leaves the company mid-negotiation | Quote remains Submitted; Admin can re-assign it or action it |
| EC-03 | Opportunity is deleted while Quote is Submitted | Quote status → `Orphaned`; Admin notified; no write-back possible |
| EC-04 | Two reps submit different Quotes at the same time | Second submission is rejected by system: "An active Quote is already pending" |
| EC-05 | SDH approves a v2 but v1 was already somehow synced | System checks `IsSyncing` on prior versions before writing back; blocks duplicate |
| EC-06 | Quote expires before SDH acts | System alerts SDH with a warning badge; approval is still allowed; Rep may be notified to extend expiry |
| EC-07 | Rep edits an unsubmitted Draft while SDH is reviewing | Not possible — Draft is locked on submission; only a new version can be created after rejection |

## 11. API Considerations (Salesforce)

The workflow uses the following Salesforce standard and custom API operations:

| Operation | Object | Trigger |
|-----------|--------|---------|
| `create()` | `Quote` | New Quote created from Opportunity |
| `update()` | `Quote` (Status, custom fields) | Each state transition |
| `create()` | `QuoteLineItem` | Line item addition |
| `update()` | `QuoteLineItem` | Line item edit |
| `create()` | `Quote` (new version) | Revision flow |
| `update()` | `OpportunityLineItem` | Price write-back on approval |
| `create()` | `OpportunityLineItem` | New line added via write-back |
| `query()` | `Quote` (by OpportunityId) | Version history retrieval |

The `CalculationStatus` field on `Quote` is monitored during price recalculation to handle async pricing states (`PriceCalculationInProgress`, `CompletedWithPricing`, `PriceCalculationFailed`).

## 12. Acceptance Criteria

| ID | Criteria |
|----|----------|
| AC-01 | A Rep can submit a Quote with at least one line item from an Opportunity in Negotiation phase |
| AC-02 | SDH receives an in-app notification within 30 seconds of submission |
| AC-03 | SDH cannot approve or reject a Quote they submitted themselves |
| AC-04 | SDH cannot reject a Quote without providing a rejection reason (UI enforced) |
| AC-05 | On approval, Opportunity line item prices match the approved Quote within 5 seconds |
| AC-06 | A rejected Quote generates a pre-populated v2 Draft with all prior line items copied |
| AC-07 | Version history shows all Quote versions in chronological order with statuses and decision timestamps |
| AC-08 | A failed write-back does not leave partial price updates in the Opportunity |
| AC-09 | No more than one Quote can be in Submitted/Pending state per Opportunity at any time |
| AC-10 | All state transitions are captured in the Quote audit trail with actor and timestamp |
